1. 背景

学院通知说我们的服务器中挖矿病毒了。服务器是实验室的主力机器，上面有十几个账户。

管理员收到报警，说我们的服务器在对外发起异常流量。于是我 SSH 进去开始排查。

4. 技术细节

攻击工具链

整个攻击用了多个组件配合：

为什么 ps 看不到进程

Linux 的进程信息来自 /proc 文件系统，ps 等工具通过读取 /proc 获取进程列表。

LKM（Linux Kernel Module）可以在内核层 hook 系统调用，拦截对 /proc 的读取请求，把指定进程从返回结果中过滤掉。由于这发生在内核层，用户层的任何工具（包括 root 权限的 ps）都无法绕过。

5. 解决方案

临时压制方案（治标）

根治方案（治本）

重装系统。 没有其他选项。

原因：LKM Trojan 已经加载进内核，而它会把自己从 lsmod 输出里删掉，无法通过 rmmod 卸载。只要内核还在，它就还在。重装会替换整个内核，才能彻底清除。

重装后加固清单：

  • MySQL 改为只监听 127.0.0.1，不对外暴露 3306
  • SSH 禁用密码登录，只允许密钥认证
  • 清理 sudoers，只保留必要的人
  • 所有账户设置强密码
  • 定期跑 rkhunter、chkrootkit

6. 反思与收获

思路对的地方

  • 第一时间查网络连接，从异常外联入手，这个方向是对的
  • 通过 VirusTotal 查 hash 来确认文件性质，比猜测快得多
  • 翻所有用户的 crontab，而不只看 root 的

走弯路的地方

最大的弯路：清理顺序搞错了。

如果顺序对了，第一次清理就能成功，不会有两天后再次中招的情况。

第二个弯路：一开始以为只是挖矿程序。

其实挖矿只是表面现象，核心是远控木马 + LKM rootkit 组成的持久化控制体系。攻击者的目的可能不只是挖矿，而是长期控制服务器。

认知偏差

我最初以为 root 权限是最高权限，有了 root 就能看到一切。但 LKM Trojan 打破了这个认知——内核层比 root 还高，运行在内核态的恶意代码可以对 root 撒谎。

这让我真正理解了 Linux 的权限模型不只是”root vs 普通用户”，还有用户态 vs 内核态这一层。

更底层的理解

这次事件让我理解了一件事：安全的信任链。

我们用 ps 排查，但 ps 依赖内核提供信息。我们用 ss 查连接，但 ss 也依赖内核。如果内核本身不可信，所有用户态的工具都不可信。

unhide 能找到隐藏进程，是因为它用了多种方法交叉验证，找出内核对不同调用返回结果的不一致性。这是一种”不信任单一来源”的思路，在安全领域非常重要。

7. 还能深入的方向

LKM rootkit 的实现原理：可以研究一下 Diamorphine、Reptile 这些开源 rootkit 是怎么 hook 系统调用的，对理解防御很有帮助。

内存取证：这次没有用到 Volatility，但它可以直接分析内存镜像找隐藏进程，不依赖内核提供信息，是比 unhide 更彻底的方案。

攻击溯源：这次最遗憾的是日志被清空了，无法确认攻击者的完整入侵路径。如果能事先部署集中式日志（把日志实时推送到另一台服务器），就算本机日志被删，还有备份可查。

10.154.48.44 这台内网机器：整个事件里最大的谜是这台内网机器。它是攻击者的跳板，还是另一台被感染的实验室服务器？这个没有查清楚，是遗留问题。学院那边什么信息都不给我提供。

自动化安全巡检：可以写一个脚本，定期检查：

  • 系统级 cron 里有没有随机命名的文件
  • /tmp、/var/tmp、/dev/shm 里有没有可执行文件
  • 有没有进程名和实际可执行文件不匹配的情况

如果要在面试里讲这个故事

我会这样讲：

“我处理过一次服务器入侵事件，最有意思的地方是它让我真正理解了 Linux 权限模型的边界。

起初以为是普通挖矿病毒，用 ps、ss 排查，但发现有网络连接却找不到对应进程。后来通过 unhide 工具发现有 70 多个隐藏进程，chkrootkit 确认是 LKM Trojan，也就是恶意内核模块在隐藏这些进程。

这让我意识到：root 权限不是最高权限，内核态才是。所有用户态工具包括 root 的 ps，都依赖内核提供信息，内核被污染后这些工具全都不可信了。

最终结论是必须重装系统，因为恶意内核模块会把自己从 lsmod 里删掉，无法通过常规手段卸载。

这次经历让我理解了安全领域一个很重要的概念：信任链。防御的核心不只是’谁有权限’，而是’你能信任谁提供的信息’。”

整个排查过程历时数天，中间经历了一次清理失败、重新发现、最终确认需要重装的完整循环。写这篇文章的目的，是希望下次遇到类似问题时，能少走一点弯路。
