# 记一次服务器被植入挖矿病毒+后门的完整排查经历

> 这篇文章记录的是一次安全事件处置过程。
---

## 1. 背景

学院通知说我们的服务器中挖矿病毒了。服务器是实验室的主力机器，上面有十几个账户。

管理员收到报警，说我们的服务器在对外发起异常流量。于是我 SSH 进去开始排查。

---

## 2. 现象与问题

打开 htop，所有 31 个 CPU 核心全部跑满，负载 64.66%。

跑了 `ps aux --sort=-%cpu`，发现占用最高的是一堆 prometheus 进程，但这些是 GitLab 自带的监控组件

看网络连接，发现：

```
tcp  ESTAB  10.176.36.22:42686  176.65.149.52:443   [kcached]
tcp  ESTAB  10.176.36.22:50150  10.154.48.44:5555   (无进程名)
tcp  ESTAB  10.176.36.22:54294  46.8.78.119:80      (无进程名)
```

1. `[kcached]` 这个进程名用方括号包着，看起来像内核线程，但内核线程不会建立外部 TCP 连接
2. 有连接存在但显示不出进程名，这不正常
3. 连接的目标都是境外 IP，端口是 443、80、5555


---

## 3. 分析过程

### 第一步：确认可疑进程的真实身份

```bash
ls -la /proc/2245/exe
# → /usr/bin/gs-dbus

ls -la /proc/32412/exe
# → /home/csh/.config/dbus/gs-dbus

cat /proc/32412/status | grep "^Uid"
# → Uid: 1015 (csh 用户)

cat /proc/32412/cmdline | tr '\0' ' '
# → mysqld   ← 伪装成 mysqld！
```

这里出现了第一个关键发现：进程名显示 `mysqld`，但实际可执行文件是 `gs-dbus`，藏在用户的 `.config/dbus/` 目录里。

### 第二步：查文件时间戳

```bash
stat /home/csh/.config/dbus/gs-dbus
# Modify: 2018-04-05   ← 伪造的
# Change: 2023-04-24   ← 真实植入时间（无法伪造）
# Access: 2025-12-23   ← 最近被执行
```

mtime 被改成了 2018 年，但 ctime 不可伪造，说明文件是 **2023 年 4 月 24 日**植入的，到被发现已经快两年。

### 第三步：查定时任务

```bash
ls /etc/cron.d/ /etc/cron.hourly/ /etc/cron.daily/ /etc/cron.monthly/ /etc/cron.weekly/
```

发现了一堆随机命名的文件：`cDuiR6Qe`、`YL6ttcWG`、`sqrs5Keb` 等，全是 1月12日创建的。

其中最关键的：
```bash
cat /etc/cron.hourly/YL6ttcWG
# → */1 * * * * root /bin/EDfKlAQo 1 1
```

每分钟以 root 身份执行 `/bin/EDfKlAQo`，这是进程复活机制。

### 第四步：查环境变量找配置

```bash
cat /proc/32412/environ | tr '\0' '\n'
# GS_ARGS=-k /home/csh/.config/dbus/gs-dbus.dat -liqD
```

有一个配置文件 `gs-dbus.dat`，内容是：
```
4T3Gud393Qf3QndVja4YSZ
```

看起来像 token，不是明文矿池地址，说明程序做了混淆处理。

### 第五步：VirusTotal 查文件 hash

这是最直接的证明方式。把 `gs-dbus` 和 `EDfKlAQo` 的 hash 上传到 VirusTotal：

- **gs-dbus**：30/63 检出，识别为 `trojan.gsnetcat`，是 gsnetcat 远控木马，会设置进程名、检测调试环境
- **EDfKlAQo**：48/65 检出，识别为 `hacktool.xhide`，是 XHide Process Faker，专门用来伪造进程名

这两个不是挖矿程序本身，而是**攻击基础设施**：一个负责远程控制，一个负责隐藏进程名。

### 第六步：翻所有用户的 crontab

```bash
for user in $(cut -f1 -d: /etc/passwd); do
  echo "=== $user ==="; crontab -u $user -l 2>/dev/null
done
```

发现了 `hzq` 用户下有一段奇怪的 cron：
```bash
@reboot /dev/shm/.cache/run > /dev/null 2>&1 & disown
@daily  /dev/shm/.cache/1
```

`/dev/shm` 是内存文件系统，重启后自动清空，攻击者故意藏在这里。

还有 `csh` 用户下有一段 base64 编码的 cron：
```bash
0 * * * * { echo L3Vzci9...|base64 -d|bash;} 2>/dev/null
```

解码后是：
```bash
/usr/bin/pkill -0 -U1015 gs-dbus 2>/dev/null || \
SHELL=/bin/bash GS_ARGS="-k /home/csh/.config/dbus/gs-dbus.dat -liqD" \
/bin/bash -c "exec -a 'mysqld' '/home/csh/.config/dbus/gs-dbus'" 2>/dev/null
```

逻辑：每小时检查 gs-dbus 是否在跑，没在跑就重启它，并伪装成 mysqld。

### 第七步：发现 rootkit

清理了一轮之后，两天后管理员又说中毒了。这时候开始怀疑有更底层的东西。

```bash
ss -antp | grep ESTAB
# ESTAB  10.176.36.22:41400  10.154.48.44:5555   (无进程名)
# ESTAB  10.176.36.22:44162  192.142.4.91:80     (无进程名)
```

连接还在，但就是找不到进程。用 iptables 封锁后，立刻出现：
```
SYN-SENT  10.176.36.22:xxxxx  10.154.48.44:5555
```

说明有进程在实时检测连接状态并重连，只是 ps 看不到它。

用 unhide 终于找到真相：

```bash
unhide proc
# Found HIDDEN PID: 1011  Cmdline: "/tmp/netools"  $PWD=/home/hzq/crond
# Found HIDDEN PID: 1012  ...
# ... (30多个 netools 进程)
# Found HIDDEN PID: 1778  Cmdline: "./-bash"  $PWD=/var/tmp/crond
# ... (60多个 -bash 进程)
```

chkrootkit 也确认了：
```
You have 72 process hidden for readdir command
You have 74 process hidden for ps command
chkproc: Warning: Possible LKM Trojan installed
```

## 4. 技术细节

### 攻击工具链

整个攻击用了多个组件配合：

```
gsnetcat (gs-dbus)
  └→ 远控木马，建立 C2 通信，伪装进程名

XHide (EDfKlAQo)
  └→ 进程名伪造工具，让挖矿程序显示为 mysqld/[kcached]

netools / -bash
  └→ 真正的恶意载荷，躲在 /home/hzq/crond 和 /var/tmp/crond

LKM Trojan
  └→ 内核模块，隐藏上述所有进程，让 ps/ss/ls 都看不到
```

### 为什么 ps 看不到进程

Linux 的进程信息来自 `/proc` 文件系统，ps 等工具通过读取 `/proc` 获取进程列表。

LKM（Linux Kernel Module）可以在内核层 hook 系统调用，拦截对 `/proc` 的读取请求，把指定进程从返回结果中过滤掉。由于这发生在内核层，用户层的任何工具（包括 root 权限的 ps）都无法绕过。

```
用户执行 ps
  → 调用 readdir("/proc")
  → 内核处理请求
  → LKM 拦截，过滤掉恶意进程的目录
  → 返回给用户的列表里没有恶意进程
```

unhide 的原理是通过多种方式交叉验证，比较系统调用结果和实际进程 ID 范围，找出不一致的地方：

```bash
unhide proc  # 通过 /proc stat 扫描
unhide sys   # 通过 getpriority() 系统调用扫描
```

### 进程名伪造原理

XHide 的核心就是调用 `execv` 时修改 `argv[0]`：

```c
execv(真实程序路径, {伪造的进程名, 参数...});
```

Linux 中 ps 显示的进程名就是从 argv[0] 读取的，所以可以显示任意名字。

chkrootkit 检测到的：
```
Checking `ps'... not infected
```

ps 本身没被改，但它读到的数据已经被内核层过滤了。

### 持久化机制（多重复活）

攻击者设置了多层复活机制，任何一层都能把程序拉起来：

```
/etc/cron.hourly/YL6ttcWG  → 每分钟 root 执行 EDfKlAQo
csh 的 crontab             → 每小时检查 gs-dbus 是否在跑
hzq 的 crontab             → 开机/每日/每月启动 /dev/shm/.cache/run
LKM Trojan                 → 保护进程不被 kill 发现
```

删一个，另一个拉起来。这也是为什么第一次清理失败的原因——当时没有按正确顺序操作，应该先断复活机制再杀进程。

## 5. 解决方案

### 临时压制方案（治标）

```bash
# 1. 先清 cron，断复活机制
crontab -u csh -r
crontab -u hzq -r
rm -f /etc/cron.d/cDuiR6Qe
rm -f /etc/cron.hourly/YL6ttcWG
rm -f /etc/cron.daily/sqrs5Keb
rm -f /etc/cron.monthly/dii9uYUH
rm -f /etc/cron.weekly/pRuydUeq

# 2. 再杀进程
kill -9 2245 32412 32411 8674
for pid in $(seq 1011 1042) $(seq 1778 1886); do
    kill -9 $pid 2>/dev/null
done

# 3. 删恶意文件
rm -f /usr/bin/gs-dbus /bin/EDfKlAQo
rm -rf /home/csh/.config/dbus/
rm -rf /var/tmp/crond/
rm -rf /home/hzq/crond/
rm -rf /dev/shm/.cache/

# 4. 封锁恶意 IP
iptables -A INPUT -s 10.154.48.44 -j DROP
iptables -A OUTPUT -d 10.154.48.44 -j DROP
iptables -A INPUT -s 192.142.4.91 -j DROP
iptables -A OUTPUT -d 192.142.4.91 -j DROP
iptables-save > /etc/iptables/rules.v4
```

### 根治方案（治本）

**重装系统。** 没有其他选项。

原因：LKM Trojan 已经加载进内核，而它会把自己从 lsmod 输出里删掉，无法通过 `rmmod` 卸载。只要内核还在，它就还在。重装会替换整个内核，才能彻底清除。

重装后加固清单：
- MySQL 改为只监听 `127.0.0.1`，不对外暴露 3306
- SSH 禁用密码登录，只允许密钥认证
- 清理 sudoers，只保留必要的人
- 所有账户设置强密码
- 定期跑 rkhunter、chkrootkit

## 6. 反思与收获

### 思路对的地方

- 第一时间查网络连接，从异常外联入手，这个方向是对的
- 通过 VirusTotal 查 hash 来确认文件性质，比猜测快得多
- 翻所有用户的 crontab，而不只看 root 的

### 走弯路的地方

**最大的弯路：清理顺序搞错了。**

第一次清理时先杀进程，再删文件，结果 cron 还在，进程马上又被拉起来。正确顺序应该是：

```
先停 cron → 再杀进程 → 再删文件
```

如果顺序对了，第一次清理就能成功，不会有两天后再次中招的情况。

**第二个弯路：一开始以为只是挖矿程序。**

其实挖矿只是表面现象，核心是远控木马 + LKM rootkit 组成的持久化控制体系。攻击者的目的可能不只是挖矿，而是长期控制服务器。

### 认知偏差

我最初以为 root 权限是最高权限，有了 root 就能看到一切。但 LKM Trojan 打破了这个认知——内核层比 root 还高，运行在内核态的恶意代码可以对 root 撒谎。

这让我真正理解了 Linux 的权限模型不只是"root vs 普通用户"，还有用户态 vs 内核态这一层。

### 更底层的理解

这次事件让我理解了一件事：**安全的信任链**。

我们用 ps 排查，但 ps 依赖内核提供信息。我们用 ss 查连接，但 ss 也依赖内核。如果内核本身不可信，所有用户态的工具都不可信。

unhide 能找到隐藏进程，是因为它用了多种方法交叉验证，找出内核对不同调用返回结果的不一致性。这是一种"不信任单一来源"的思路，在安全领域非常重要。

## 7. 还能深入的方向

**LKM rootkit 的实现原理**：可以研究一下 Diamorphine、Reptile 这些开源 rootkit 是怎么 hook 系统调用的，对理解防御很有帮助。

**内存取证**：这次没有用到 Volatility，但它可以直接分析内存镜像找隐藏进程，不依赖内核提供信息，是比 unhide 更彻底的方案。

**攻击溯源**：这次最遗憾的是日志被清空了，无法确认攻击者的完整入侵路径。如果能事先部署集中式日志（把日志实时推送到另一台服务器），就算本机日志被删，还有备份可查。

**10.154.48.44 这台内网机器**：整个事件里最大的谜是这台内网机器。它是攻击者的跳板，还是另一台被感染的实验室服务器？这个没有查清楚，是遗留问题。学院那边什么信息都不给我提供。

**自动化安全巡检**：可以写一个脚本，定期检查：
- 系统级 cron 里有没有随机命名的文件
- `/tmp`、`/var/tmp`、`/dev/shm` 里有没有可执行文件
- 有没有进程名和实际可执行文件不匹配的情况

## 8. 可复用清单

### 快速排查命令

```bash
# 查 CPU 占用最高的进程
ps aux --sort=-%cpu | head -20

# 查所有外部建立的连接（含进程名）
ss -antp | grep ESTAB

# 查进程的真实可执行文件
ls -la /proc/<PID>/exe

# 查进程的运行用户
cat /proc/<PID>/status | grep "^Uid"

# 查进程的环境变量
cat /proc/<PID>/environ | tr '\0' '\n'

# 查文件真实时间戳
stat <文件路径>

# 查所有用户的 crontab
for user in $(cut -f1 -d: /etc/passwd); do
  echo "=== $user ==="; crontab -u $user -l 2>/dev/null
done

# 查 /tmp /var/tmp /dev/shm 里的可执行文件
find /tmp /var/tmp /dev/shm -type f -executable 2>/dev/null

# 查隐藏进程
apt install unhide -y && unhide proc && unhide sys

# 查 rootkit
apt install chkrootkit rkhunter -y
chkrootkit
rkhunter --check --skip-keypress

# 查文件 hash
md5sum <文件> && sha256sum <文件>
```

### 清理顺序 Checklist

```
□ 1. 先清所有 cron（系统级 + 所有用户）
□ 2. 杀恶意进程
□ 3. 删恶意文件
□ 4. 封锁恶意 IP（iptables）
□ 5. 修改所有账户密码
□ 6. 检查 sudoers，移除不必要的权限
□ 7. 检查 SSH authorized_keys
□ 8. 验证清理结果（unhide、ss）
□ 9. 保存取证证据
```

### 封锁 IP 模板

```bash
# 彻底封锁某个 IP 的所有往来流量
iptables -A INPUT -s <IP> -j DROP
iptables -A OUTPUT -d <IP> -j DROP
iptables-save > /etc/iptables/rules.v4
```

### 快速验证是否还有外联

```bash
# 监控新建立的外部连接，每5秒刷新
watch -n 5 "ss -antp | grep ESTAB | grep -v '127.0.0\|内网网段'"
```

## 9. 结构化知识沉淀

### 这个问题属于哪个知识域

主要横跨：
- **Linux 系统安全**：权限模型、进程管理、cron 机制
- **网络安全 / 应急响应**：入侵检测、取证、清除
- **内核与操作系统原理**：用户态 vs 内核态、系统调用、LKM 机制

### 触发的核心知识模型

**Linux 权限模型**：
```
用户态（ring 3）：root 是最高，但仍受内核限制
内核态（ring 0）：内核代码，LKM 运行在这里
```

**信任链模型**：
```
工具可信 → 依赖内核可信 → 内核被污染则工具不可信
```

**攻击持久化模型**：
```
多层复活机制 = cron + 内存驻留 + 内核保护
清除时需要同时断掉所有层，否则会复活
```

### 如果要在面试里讲这个故事

我会这样讲：

> "我处理过一次服务器入侵事件，最有意思的地方是它让我真正理解了 Linux 权限模型的边界。
>
> 起初以为是普通挖矿病毒，用 ps、ss 排查，但发现有网络连接却找不到对应进程。后来通过 unhide 工具发现有 70 多个隐藏进程，chkrootkit 确认是 LKM Trojan，也就是恶意内核模块在隐藏这些进程。
>
> 这让我意识到：root 权限不是最高权限，内核态才是。所有用户态工具包括 root 的 ps，都依赖内核提供信息，内核被污染后这些工具全都不可信了。
>
> 最终结论是必须重装系统，因为恶意内核模块会把自己从 lsmod 里删掉，无法通过常规手段卸载。
>
> 这次经历让我理解了安全领域一个很重要的概念：信任链。防御的核心不只是'谁有权限'，而是'你能信任谁提供的信息'。"
